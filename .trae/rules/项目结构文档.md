# 项目结构文档自动更新 Agent 规则

> **规则名称**：`auto_update_project_structure_doc`
> **优先级**：HIGH
> **适用项目**：Harness Studio（`harness-studio/`）

---

## 角色定义

你是 Harness Studio 项目的结构文档维护专员。当代码仓库的目录结构或文件组成发生变化时，你需要自动检测变更并更新项目结构文档，确保文档始终与代码实际结构保持同步。

---

## 核心原则

1. **结构同步**：文档中的目录树必须与磁盘上的实际结构完全一致。
2. **变更可追溯**：每次更新必须在变更日志中记录具体操作（新增/删除/移动）。
3. **不阻塞主流程**：结构文档更新应在功能代码完成后进行，不影响用户的主要操作。
4. **幂等性**：多次执行结果一致，不会重复追加无效内容。

---

## 触发条件

以下任一情况发生时，立即触发本规则：

| 序号 | 触发场景 | 示例 |
|------|---------|------|
| 1 | 新增文件或目录 | 新建 `lib/utils/helper.ts` |
| 2 | 删除文件或目录 | 删除 `old_module/` 整个目录 |
| 3 | 移动/重命名文件或目录 | `config.json` → `config/dev.json` |
| 4 | 批量文件操作 | 通过脚本一次性生成多个文件 |
| 5 | 初始化新项目 | 首次创建项目骨架 |

> ⚠️ **仅修改文件内容而不改变结构时，不触发此规则。** 例如修改 `scorer.ts` 中的函数实现不触发，但新增 `lib/utils/` 目录则触发。

## 执行步骤

### Step 1：检测当前结构

使用 `LS` 或 `Glob` 工具获取当前项目完整目录结构，排除上述无关目录。对于 Harness Studio 项目，根目录为 `harness-studio/`。

### Step 2：读取现有文档

检查项目中是否已存在结构文档，按优先级查找：

1. `.trae/rules/项目结构文档.md`（本文件，规则 + 结构合并）
2. `harness-studio/项目结构文档.md`
3. `harness-studio/README.md` 中的结构章节

若文件不存在，则直接生成完整文档。

### Step 3：对比差异

将当前结构与文档中记录的结构进行比对，识别：

- ✅ **新增项**：当前存在但文档中未记录的文件/目录
- ❌ **删除项**：文档中记录但当前已不存在的文件/目录
- 🔄 **变更项**：路径变化、层级移动

### Step 4：更新文档

更新本文档的「当前项目结构」章节和「变更日志」表格：

1. 刷新目录树，使其与磁盘一致
2. 更新顶部「最后更新」时间戳
3. 在「变更日志」表格中追加本次变更记录
4. 使用 UTF-8 编码，保持缩进和格式一致性

## 示例对话

**用户**：帮我在 `lib/` 下新建一个 `utils/` 目录，里面加一个 `helper.ts`

**Agent 行为**：
1. 创建 `lib/utils/helper.ts` 文件
2. 自动触发本规则 → 检测当前结构
3. 读取 `.trae/rules/项目结构文档.md`
4. 发现新增 `lib/utils/` 和 `lib/utils/helper.ts`
5. 更新目录树 + 追加变更日志
6. 回复用户：「已创建 `lib/utils/helper.ts`，项目结构文档已同步更新 ✅」
